### 一、流式输出（Streaming）的技术原理

#### 1.1 定义与抽象模型
流式输出是一种**数据传输模式**，其核心特征在于：发送端将待传输的数据集分割为一系列有序的、尺寸较小的数据块（Chunks），并以连续或近似连续的方式逐块推送至接收端。接收端无需等待完整数据集的传输结束，即可对已到达的数据块执行解析、渲染或处理操作。

#### 1.2 Stream VS SSE
**“Stream”（流）是一种抽象的模式，而“SSE”是实施这种模式的一个具体工具。**

**1. “Stream”（流）是一种行为方式**
-定义：把一大块数据切成一小块一小块，**像流水一样连续不断地**发送出去。

**2. “SSE”是一种具体的运输工具**
-定义：一种基于 HTTP 协议的、专门用来从**服务器单向推送给网页**的技术规范。

### 二、MCP
**MCP 协议新增了 `Streamable HTTP` 作为更推荐的传输方式，用以解决原生 `SSE` 在双向通信和长连接稳定性上的局限性。** 这不是“弃用”，而是“技术升级和替代”。

#### 1.1 为什么 MCP 当初要用 SSE？

MCP（Model Context Protocol，模型上下文协议）刚推出时，主要想让**云端 AI （比如 Claude Desktop）** 能调用**你本地电脑上跑的工具**。

*   **场景**：AI 在云端，数据库/文件在你笔记本电脑上。
*   **需求**：云端 AI 要能**主动**把任务发给本地电脑，本地电脑执行完了再把结果**返回**给云端。
*   **方案**：因为电脑没有公网 IP，云端没法直接连接，所以 MCP 最初选择了 **SSE + JSON-RPC**。
    *   本地客户端（Client）主动向云端服务器（Server）发起一个 HTTP 连接。
    *   连接建立后，**服务器就可以通过这根水管（SSE）源源不断地向客户端推送指令**。
    *   客户端如果要回复服务器，走的是**另一条路**（单独的 HTTP POST 请求）。

#### 2. SSE 在 MCP 场景下暴露出的问题

SSE 是“服务器→客户端”的单行道。

| 问题 | 解释 | 后果 |
| :--- | :--- | :--- |
| **必须维持长连接** | 服务器要随时给客户端派活，**SSE 连接必须一直保持打开**。 | 网络稍微波动一下，连接断了，服务器就找不到客户端了，任务直接失败。 |
| **回复走弯路** | 客户端干完活要回信，**不能走 SSE 原路返回**，得重新打个 HTTP 请求回去。 | 服务器得专门维护一个 **“连接 ID 和 回复地址”** 的对照表，逻辑复杂，容易错乱。 |
| **不扛高并发** | 每个客户端都要占着服务器一个长连接不放。 | 如果同时有 1000 个人在用，服务器就要维持 1000 个一直开着的连接，**资源消耗巨大**。 |

#### 3. `Streamable HTTP` 是怎么解决这些问题的？

2025 年，MCP 协议官方（Anthropic）推出了 **`Streamable HTTP`** 传输方式。它的设计更聪明：

*   **做法**：客户端和服务器**不再死守着一根水管**。
    *   客户端发一个普通 HTTP 请求给服务器。
    *   服务器如果需要连续返回很多数据（比如 AI 的回答），就在**同一个 HTTP 响应**里用 **分块传输（Chunked Encoding）** 一点一点推回去。
    *   推完了，**连接就可以挂断**。
    *   下次要发新指令，**再重新建立连接**。

*   **好处**：
    1.  **双向通道**：请求和响应都在同一个 HTTP 回合里完成，逻辑干净。
    2.  **无需维持长连接**：随用随连，不占着茅坑不拉屎。服务器压力小，网络抖动也不怕。
    3.  **基础设施友好**：普通的 HTTP 负载均衡、网关、CDN 都能完美支持，不需要特殊配置。

### 项目选择SSE的原因
#### 1. 技术栈的自然契合

- **FastAPI 原生支持**：`StreamingResponse` + `media_type="text/event-stream"` 是 FastAPI 官方推荐的流式输出范式，无需额外依赖或协议升级。
- **LangGraph + 通义千问的流式特性**：大模型生成过程天然是逐 token 产出，SSE 可以逐块推送，与生成器（`yield`）无缝衔接。

---

#### 2. 通信模式的精确匹配

您的业务核心是 **“服务器单向向浏览器推送 AI 生成的回答”**，这一模式完全符合 SSE 的设计定位：
- **单向性**：SSE 专为服务器→客户端推送而生，与需求高度一致。
- **无反向复杂交互**：不需要通过同一连接向服务器发送控制指令（如中断、重新生成），因此无需 WebSocket 的双向通道。

---

#### 3. 与 WebSocket 的对比优势

| 维度 | SSE | WebSocket |
|------|-----|------------|
| **协议开销** | 基于 HTTP，无需额外握手 | 需独立协议升级（`101 Switching Protocols`） |
| **实现复杂度** | 后端 `StreamingResponse`，前端 `EventSource`，代码量极少 | 需管理连接生命周期、心跳、重连逻辑 |
| **自动重连** | 浏览器内置重连机制 | 需手动实现 |
| **适用性** | 恰好覆盖“推送文本”场景 | 过度设计，引入不必要复杂性 |

因此，使用 SSE 避免了 WebSocket 带来的“大炮打蚊子”问题。

---

#### 4. 可观测性与调试友好

- **HTTP 层可见**：SSE 请求在浏览器 Network 面板中以普通 HTTP 请求呈现，`Content-Type: text/event-stream` 一目了然。
- **消息格式标准**：每条数据以 `data:` 前缀 + 双换行结尾，易于抓包分析和日志审计。

---

### 结论
**技术栈（FastAPI + 流式生成）、通信模式（单向推送）、工程复杂度（低）与协议特性（自动重连、HTTP 兼容）** 
这一选型符合“最小必要复杂度”原则。
